# Refferq hardening plan

## Scope
Только сервис `refferq` (без изменения корневого репозитория).

## Подготовка перед началом hardening

### 0) Что уже зафиксировано (чтобы не забыть)
- `/api/auth/login` и `/api/auth/register` **нельзя удалять**, они нужны как entrypoint для email-only UX/интеграций, но в них больше нет проверок пароля.
- Любой payload, который пытается передать password/credentials, должен быть отклонён с `410 Gone` и нейтральным сообщением.
- Логика по статусу:
  - `ACTIVE` + `ADMIN`/`AFFILIATE` — рабочие для входа/работы API.
  - `PENDING` — для `/api/auth/login`/`register` может продолжать OTP-поток, если это существующий пользователь (без входа в защищённые зоны до апрува), `/api/auth/verify-otp` и защищённые маршруты всегда дают 403.
  - `INACTIVE`/`SUSPENDED` — только нейтральный `403`, OTP не отправляем.
- Для несуществующего email:
  - `/api/auth/login` → `202` + `next: "register"` без отправки OTP (чтобы не раскрывать факт существования).
  - `/api/auth/register` + валидный новый email + name → создаём `PENDING`-пользователя и шлём OTP.
- `role` в `/api/auth/register` не должна давать привилегий по умолчанию: по контракту мы принимаем только intended-role `AFFILIATE` (админскую активацию не запускаем через публичный flow).
- На этом этапе по авторизации и статусам критических неоднозначностей не осталось.

### 0.1) Финальное решение по обсуждению
- `login` и `register` не удаляем: они остаются как public OTP entrypoints.
- Password-based схема убрана из публичных входов, но endpoint-ы остаются для совместимости интеграций.
- `register` для существующего пользователя:
  - `PENDING`/`ACTIVE` → `202 + next: "otp"` и отправка OTP;
  - `INACTIVE`/`SUSPENDED` → `403` без отправки OTP.
- `login` для неизвестного email всегда отвечает `202 + next: "register"` без отправки OTP.
- Для protected/API маршрутов:
  - `PENDING`/`INACTIVE`/`SUSPENDED` → 403;
  - `401` только при отсутствии/невалидном токене.

### 0.2) Текущее состояние hardening
- ✅ `auth` flow: passwordless entrypoints + unified статусная логика.
- ✅ `getProgramSettings` вынесен в helper и используется для `settings`/`auto payout`/`webhook holdDays`.
- ✅ Идемпотентность добавлена для `track` и `webhook` conversion.
- ✅ `/admin/payouts/auto` переведён на транзакционную обработку на уровне списка комиссий.
- ⚠️ Остаётся пройтись по шагам тестирования и зафиксировать smoke-скрипты в репо до merge.

### 0.3) Ответы на открытые вопросы
- `login` и `register` не удаляем. Они остаются публичными OTP entrypoints:
  - `/api/auth/login` используется для входа по email.
  - `/api/auth/register` используется для приглашения новых аккаунтов и повторного входа по существующему email через OTP.
- Password-based логика для внешних интеграций убирается: любые payload с credential-полями (`password`, `otp`, и т.п.) на публичных auth-роутах отбрасываются как устаревшие.
- Для нового пользователя в `register` пока используется placeholder-пароль, чтобы не ломать схему (`password` в Prisma остаётся обязательным до миграции); по-хорошему парольную колонку стоит перевести в optional/nullable поэтапно.
- По шагам hardening доработки выполняются: сначала исправления auth/status, потом payout/трекинг, затем тестирование и smoke.

### 0.4) Что должно быть закрыто до старта hardening-сессии
- Утвердить, что `JWT_SECRET` реально живёт в окружении сервера (и валидируется один раз при старте), а не подменяется локально.
- Проверить матрицу статусов:
  - `ACTIVE` + роль (`ADMIN`/`AFFILIATE`) — нормальный доступ.
  - `PENDING` — без доступа в защищённые зоны, OTP разрешён.
  - `INACTIVE`/`SUSPENDED` — только нейтральный `403`, OTP/Protected API запрещены.
- Согласовать с командой временную стратегию по полю `password`:
  - сейчас используется placeholder-хэш для новых пользователей;
  - в следующем тикете снять обязательность `password` из БД или отдельным скриптом сделать безопасный backfill/migration.
- Зафиксировать порядок запуска и отката:
  - дамп БД перед шагами с выплатами;
  - журнал инцидентов/изменений в `HARDENING.md` после каждого шага.
- Нужный baseline для проверки: `npm run build` должен проходить при валидном `.env` (в т.ч. `JWT_SECRET >= 32`).

### 1) Входные решения и риски
- Password-based auth полностью убираем из рабочей логики.
- `/api/auth/login` и `/api/auth/register` остаются compatibility-путями для OTP-flow (чтобы не ломать внешние интеграции), но **не работают с паролями**:
  - `POST /api/auth/login` — принимает `email` и инициирует/повторяет отправку OTP (passwordless).
  - `POST /api/auth/register` — принимает `email` (+ `name`, `role`), создаёт/обновляет аккаунт и запускает OTP.
  - Любой password-based payload на этих маршрутах отклоняется как устаревший (`410 Gone`) с нейтральным сообщением без утечки деталей.
- Поведение ролей: `ADMIN` и `AFFILIATE` валидируются через JWT+DB-check; `PENDING/INACTIVE/SUSPENDED` всегда получают 403/401 согласно политике «учётная запись неактивна». 
- `/api/auth/me` — источник истины для состояния сессии после `verify` и проверки токена в middleware.

### 2) Безопасность и окружение
- Проверить и зафиксировать единый `JWT_SECRET` (минимум 32 символа), убрать «ленивые» `!` и добавить валидацию запуска.
- Подготовить локально/на стенде отдельный `DATABASE_URL`, отдельный `.env` и зафиксировать seed-команды.
- Прописать fallback-поведение для отсутствующих секретов: fail-fast при запуске критичных сервисов.

Рекомендуем:
- Добавить `JWT_SECRET`-guard в `lib/auth-config.ts` (или аналог) и валидировать его во время startup.
- Единый стандарт ошибок: `401` — отсутствует/невалидный токен, `403` — токен валиден, но прав недостаточно или пользователь неактивен.

Ключевые договоренности после обсуждения:
- `/api/auth/login` и `/api/auth/register` **сохраняются** как точки входа OTP (поддержка legacy интеграций, но без парольной логики).
- Любые payload-ключи под аутентификацию (`password`, `passcode`, `otp`, `token` и т.п.) -> `410 Gone`.
- `role` в публичном `register` не повышает права: итоговый intent только `AFFILIATE`.
- Для неизвестного email в `login` возвращаем `202` + `next: "register"` без `sendOTP`; `register` всегда выполняет OTP-flows.
- Технически на текущую схему (`User.password` обязательный в Prisma) passwordless для новых пользователей сейчас = заглушка/генерация поля password без верификации; при следующем окне миграции — сделать поле nullable/необязательным и убрать это полностью.

### 3) Данные и откат
- Подготовить дамп базы (или транзакционный бэкап) перед любыми шагами payout/балансов.
- Зафиксировать SQL-список миграций и «снимок» текущих сумм балансов/статусов payout/commission.
- Определить план отката для каждого шага (что откатываем руками, что via git/rollback).

### 4) Тестовый каркас
- До изменений добавить/подтянуть базовый smoke-инструментарий (минимум для auth/track/payout/api invariants), чтобы шаги 10–11 можно было мерить не «на глазок».
- Зафиксировать тестовые пары: валидные/истёкшие/невалидные JWT, корректные и некорректные payload-и track/webhook, сценарий payout happy-path и rollback.

Рекомендуемый минимальный контракт до старта:
- `/api/auth/me`: валидный токен / просроченный / подменённый.
- `/api/track/conversion`: валидация `referralCode`, `amountCents` (или `amount`), `eventType`, `orderId`, `customerEmail`.
- payout/manual: happy-path + искусственная ошибка после первой критичной записи.
- webhook: `Idempotency-Key`/`eventId` с повторной доставкой (idempotent repeat must be safe).
- Рекомендуемый тест-инструмент: `vitest` + `tsx` smoke-скрипт против поднятого dev-сервера (без усложнения CI/fixture-системы на первом проходе).

### 5) Окно выполнения
- Выполнять hardening по одному шагу за коммит, с остановкой при любых нарушенияных тестов.
- Каждому шагу давать: цель, критерии «done», и `git`-чекпоинт.

### 6) Мой рекомендуемый финальный контракт (фикс для старта)

- **`/api/auth/login`**
  - Passwordless-only, принимает `{ email }`.
  - Если пришёл `password` или другие legacy-поля для пароля → `410 Gone` с нейтральным сообщением.
  - Корректный email → `202 Accepted`, возвращаем `next: "otp"` и запускаем/обновляем OTP.
  - Если пользователя нет → `202 Accepted`, возвращаем `next: "register"` без отправки OTP, чтобы не раскрывать существование email.

- **`/api/auth/register`**
  - Passwordless-only, принимает `{ email, name?, role? }`.
  - Если email новый: создаём пользователя (пока с placeholder-паролем, пока поле `password` остаётся обязательным), статус `PENDING`, создаём/обновляем affiliate-данные при роли `AFFILIATE`, отправляем OTP, `202 Accepted` с `next: "otp"`.
  - Если email уже существует:
    - `active`/`pending` — возвращаем `202` с `next: "otp"` и отправляем OTP для входа, чтобы не ломать UX существующего пользователя.
    - `inactive`/`suspended` — возвращаем `403` (нейтрально), OTP не отправляем.

- **Auth статусы и middleware**
  - `PENDING/INACTIVE/SUSPENDED` всегда `403` в защищённых API и страницах (не-`admin`/не-`affiliate` доступы).
  - `401` только для отсутствующего/невалидного/просроченного токена.

- **Tracking/Webhook идемпотентность**
  - Канал идемпотентности: в первую очередь `X-Idempotency-Key`, если не передан — `eventId` из payload.
  - Схема: хранить `integration_id + idempotency_key` как уникальный ключ и возвращать ранее обработанный результат при повторе.
  - Для `/api/track/conversion` и webhook обязательные поля в валидации:
    - `referralCode`, `eventType`, `customerEmail`, (`orderId` или `eventId`), сумма (`amount` или `amountCents`), `currency`.

## Шаги выполнения (коммит после каждого шага)

1. **Шаг 1 — Проверить и зафиксировать текущую аутентификацию/права**
   - Прогнать статический и runtime-аудит auth-маршрутов и middleware.
   - Убедиться, что `JWT_SECRET` валидируется централизованно и не используется «с голого» `!`.
   - Проверка: тесты/смоук для `/api/auth/me` с валидным, просроченным и невалидным токеном.

2. **Шаг 2 — Удалить парольную схему, оставить только OTP**
   - Сделать `/api/auth/login` и `/api/auth/register` passwordless-совместимыми endpoint'ами для OTP.
   - Удалить password-hash/compare и случайную генерацию паролей из auth flow.
  - Зафиксировать ответ по legacy-поводу: password-based payload => `410`, корректный email-only payload => `202` с переходом в OTP flow.
   - Проверка: тесты на /api/auth/login и /api/auth/register — отклонение старого password-based сценария и успешный passwordless-сценарий.

3. **Шаг 3 — Сделать payout атомарным**
   - Перевести payout-цепочку в single Prisma transaction: создание записи + списание/начисление + статусные переходы.
   - Убедиться, что бизнес-ошибки/исключения не оставляют промежуточного состояния.
   - Проверка: unit/integration тест на happy path и rollback при ошибке на 2+ шагах.

4. **Шаг 4 — Защитить от частичной записи**
   - Для payout-сценариев и связанных сервисных операций добавить явный rollback-паттерн (transaction / компенсацию).
   - Исключить сценарии с «half-committed» бизнес-состоянием при сбое фоновых/batch задач.
   - Проверка: тест с искусственным падением после первого шага + контроль целостности таблиц.

5. **Шаг 5 — Оставить OTP как основной флоу входа/подтверждения**
   - Уточнить, что OTP используется и в UI, и в API, и в fallback-поведения.
   - Проверить все состояния ошибок и rate-limit для OTP.
   - Проверка: тесты на send/verify OTP и ограничения по частоте.

6. **Шаг 6 — Добавить валидацию и идемпотентность в tracking-ядро**
   - Нормализовать валидаторы входных данных для `/api/track/conversion` и webhook-роутов.
   - Ввести идемпотентный ключ для повторной доставки событий (или корректно использовать заказной ключ).
   - Проверка: тесты на невалидные payloads + повторные события с тем же ключом.

7. **Шаг 7 — Привязать константы/параметры к базе или env**
   - Удалить локальные «магические» дефолты в логике бизнес-правил.
   - Проверить, что ключевые значения берутся из `settings`/БД и документируются.
   - Проверка: тест инициализации с отсутствием optional-значений.

8. **Шаг 8 — Привести валюту к единому виду**
   - Свести символ в одном стиле (`$`) на уровне форматтера.
   - Проверить все UI/отчёты на единообразный вывод.
   - Проверка: unit тесты на форматирование денежных значений.

9. **Шаг 9 — Довести flow до простого и понятного**
   - Упростить логику click/referral без усложнений: явные состояния визита, конверсии, лид-ивента.
   - Убедиться, что сценарии воспроизводимы и декомпозированы.
   - Проверка: тесты по роуту `/r/[code]` и триггерам лида.

10. **Шаг 10 — Ужесточить типизацию (убрать `any` в критических участках)**
    - Пройтись по payout/auth/OTP/tracking и заменить `as any` там, где это влияет на надежность.
    - Проверка: `pnpm`/`npm test` + `tsc` без деградации типов.

11. **Шаг 11 — Добавить тестовое покрытие по ядру**
    - Зафиксировать базовый набор юнит-тестов: auth, tracking, деньги/балансы, payout, API-инварианты.
    - Ввести минимальные smoke и regression-тесты для критичных сценариев.
    - Проверка: прогон CI-команд для треков и core API (без выноса в отдельные эксперименты).

### 12) Актуальный статус hardening (текущий цикл)

- `npm run build` (с `JWT_SECRET=...`) — выполняется.
- Для live-чека рекомендуемые переменные: 
  - `JWT_SECRET` (минимум 32)
  - `DATABASE_URL` (иначе rate-limit/Auth endpoints деградируют в 429, т.к. завязаны на БД)
- `npm run hardening:smoke` — выполняет:
  - чистые проверки контрактов (8 checks, all pass),
  - live API проверки при доступном `HARDENING_BASE_URL`, `DATABASE_URL` и `JWT_SECRET` (скрипт пропускает их с `⚪` если API недоступно).
